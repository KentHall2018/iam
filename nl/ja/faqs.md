# FAQ
{: #iamfaq}

## {{site.data.keyword.cloud_notm}} ID およびアクセス管理とは? 
{: #whatisiam}

ID およびアクセス管理 (IAM) を使用すると、プラットフォーム・サービスのユーザーをセキュアに認証でき、また {{site.data.keyword.cloud_notm}} プラットフォーム全体でリソースへのアクセスを制御できます。一部の IBM Cloud サービスは、Cloud IAM を使用したアクセス制御に対応しています。それらのサービスは、ユーザーが一度に複数のリソースに素早く簡単にアクセスできるように、アカウント内でリソース・グループに編成されます。ご使用のアカウント内のリソースへのアクセス権限をユーザーおよびサービス ID に割り当てるために、Cloud IAM アクセス・ポリシーが使用されます。 詳しくは、[{{site.data.keyword.cloud_notm}}ID およびアクセス管理](/docs/iam/index.html#iamoverview)を参照してください。

## IAM 対応サービスとは?
{: #iam-enabled}

IAM 対応サービスはリソース・グループ内になければならず、サービスへのアクセス権限は IAM アクセス・ポリシーを使用して付与されます。カタログから IAM 対応サービスを作成する際、それをいずれかのリソース・グループに割り当てる必要があります。詳しくは、『[リソースとは](/docs/resources/acct_resources.html#resource)』を参照してください。

{{site.data.keyword.containerlong_notm}} は唯一の例外であり、IAM アクセス制御下にありますが、常にデフォルト・リソース・グループに割り当てられます。したがって、カタログから作成するときにリソース・グループを選択するオプションはなく、他のリソース・グループに割り当てることはできません。


## IAM と Cloud Foundry は関連していますか?
{: #iam-cloudfoundry}

これらは関連していません。Cloud Foundry は、組織、スペース、および Cloud Foundry 役割を使用してアクセス管理を行うオープン・ソース・プラットフォームです。IAM は、プラットフォーム・サービスのユーザーをセキュアに認証するため、および、リソース・グループおよび IAM アクセス役割を使用して {{site.data.keyword.cloud_notm}} プラットフォーム全体でリソースへのアクセスを制御するための、安全な方法です。

これらのアクセス管理システムはまったく異なります。IAM リソースはリソース・グループに属し、Cloud Foundry リソースは組織およびスペースに属します。IAM アクセス・ポリシーが、リソース・グループ内のリソースへのアクセス権限を付与するために使用されます。一方、Cloud Foundry 組織役割とスペース役割が、スペース内の Cloud Foundry リソースへのアクセス権限をユーザーに付与するために使用されます。Cloud Foundry スペース内のものに対するアクセス権限を IAM が付与することはなく、リソース・グループ内のリソースへのアクセス権限を Cloud Foundry 役割が付与することはできません。

## 自身のアクセス権限を確認するにはどうすればよいですか?
{: #iam-access}

**「管理」** &gt; **「アクセス (IAM)」**と進み、「ユーザー」列で自分の名前を選択します。次に、確認したいアクセス権限に応じて、以下のタブを開きます。

* どのアクセス・グループにいるのかを判別するには、**「アクセス・グループ」**を選択します。
* IAM アクセスの場合は、**「アクセス・ポリシー」**を選択します。
* Cloud Foundry 役割の場合は、**「Cloud Foundry アクセス権限」**を選択します。

## リソースへのアクセス権限を要求するにはどうすればよいですか?
{: #request-access}

アカウント所有者は、アカウント内の任意のリソースへのアクセス権限を更新できます。あるいは、サービスまたはサービス・インスタンスの管理者役割が割り当てられている任意のユーザーに連絡することもできます。 

## リソース・グループとアクセス・グループを使用する必要があるのはなぜですか? 
{: #resource-groups}

リソース・グループは、リソースの論理コンテナーです。リソースは、作成されるといずれかのリソース・グループに割り当てられ、追加後に移動することはできません。 

アクセス・グループは、アクセス権限の割り当てが容易になるようにユーザーおよびサービス ID のセットを単一のエンティティーに簡単に編成するために使用されます。アクセス・グループにポリシーを 1 つ割り当てるだけで、すべてのメンバーにそれらのアクセス権限を付与することができます。複数のユーザーまたはサービス ID が同じアクセス権限を必要とする場合は、個々のユーザーまたはサービス ID ごとに同じアクセス権限を複数回割り当てる代わりに、アクセス・グループを作成してください。

リソース・グループとアクセス・グループの両方を使用すると、限られた数のポリシーを割り当てることでアクセス権限割り当てポリシーを簡素化できます。ユーザーおよびサービス ID の特定のグループがアクセス権限を必要とするすべてのリソースを単一のリソース・グループに編成し、それらのユーザーおよびサービス ID をすべてまとめて 1 つのアクセス・グループにし、次に、リソース・グループ内のすべてのリソースへのアクセス権限を付与する単一のポリシーを割り当てることができます。

## ユーザーがリソース・グループ内にリソースを作成できるようにするにはどうすればよいですか?
{: #resources}

リソース・グループ内にリソースを作成するには、ユーザーには 2 つのアクセス・ポリシーが必要です。1 つはリソース・グループ自体に割り当てられたもので、もう 1 つはグループ内のリソースに割り当てられたものです。リソース・グループ自体へのアクセス権限は、単に、リソースが編成されているコンテナーへのアクセス権限であり、このタイプのポリシーは、ユーザーがグループへのアクセス権限を表示、編集、または管理することを許可しますが、グループ内のリソースは対象ではありません。リソース・グループ内のサービスへのアクセス権限により、ユーザーはサービス・インスタンスを処理できるようになります。これは、ユーザーがサービス・インスタンスを作成できることを意味します。

したがって、最小でもユーザーは以下のアクセス権限を持っている必要があります。

* リソース・グループ自体に対するビューアー以上の役割
* リソース・グループ内の当該サービスまたはすべてのサービスに対するエディター以上の役割


## 他の人にアクセス権限を付与するために必要なアクセス権限は何ですか?
{: #user-access}

ユーザーにアクセス権限を割り当てたいサービスまたはリソースに対する管理者役割を持っている必要があります。アカウント内のすべてのサービスまたはリソースへのアクセス権限を割り当てたい場合は、管理者役割と共に、ID およびアクセス管理に対応するすべてのサービスに関するポリシーが必要です。 

## リソース・グループを管理するためのアクセス権限の付与とリソース・グループ内のリソースへのアクセス権限の付与との違いは何ですか?
{: #providing-access}

リソース・グループを管理するためのアクセス権限を持っている場合、割り当てられた役割に応じて、リソース・グループ自体のアクセス権限の表示、名前の編集、および管理を行うことができます。リソース・グループ自体へのアクセス権限が、そのグループ内のリソースへのアクセス権限をユーザーに付与するわけではありません。

リソース・グループ内のリソースへのアクセス権限を持っている場合、割り当てられた役割に応じて、インスタンスの編集、削除、および作成を行ったり、リソース・グループ内の指定されたサービスについてのすべての管理アクションを行ったりできます。 

例えば、アカウント管理サービスのプラットフォーム管理役割とアクションについては、[プラットフォーム役割の表](/docs/iam/users_roles.html#platformrolestable2)を参照してください。

## ユーザーを削除できるのは誰ですか?
{: #remove-users}

ユーザーを削除できるのはアカウント所有者だけです。 

## 多要素認証をオンにするにはどうすればよいですか?
{: #multi-factor}

**「管理」** &gt; **「アクセス (IAM) (Access (IAM))」**に移動して、**「設定」**を選択します。 
**「多要素認証 (Multi-factor authentication)」**を選択し、 **「変更の適用」**をクリックします。 詳しくは、[多要素認証の有効化](/docs/iam/mfa.html#enablemfa)を参照してください。

## サービス役割とプラットフォーム役割の違いは何ですか? 
{: #service-platform-roles}

サービス役割とプラットフォーム役割は、2 つの異なるタイプの役割です。 

* プラットフォーム役割は、アカウント内のサービスに対してどのような処理を行うのか (例えば、インスタンスの作成、インスタンスのバインド、サービスへのユーザーのアクセス権限の管理) ということです。プラットフォーム・サービスに対して、これらの役割は、ユーザーが、例えばリソース・グループの作成やサービス ID の管理などを行うことを可能にします。プラットフォーム役割は、管理者、エディター、オペレーター、およびビューアーです。 

* サービス役割は、サービスに対してアクションを実行できるかどうかを定義し、API 呼び出しの実行や UI へのアクセスなどの各サービスに固有です。サービス役割は、管理者、ライター、およびリーダーです。これらの役割の適用方法について詳しくは、特定のサービスの資料を参照してください。

## リソース・グループと Cloud Foundry の組織およびスペースとの違いは何ですか?
{: #groups-organizations}

{{site.data.keyword.Bluemix_notm}} の開始時には、アクセス制御とリソース編成のための Cloud Foundry という名前のオープン・ソース・プラットフォーム・サービスが、リソースへのアクセス権限の編成と制御のための単一の方法でした。{{site.data.keyword.Bluemix_notm}} が拡張されたときに、すべてのタイプのサービスおよびリソースで使用できる新しい方法が必要になりました。そこで、さまざまなタイプのリソースをグループ化および編成するためにリソース・グループが使用されるようになり、サービスおよびリソースへのアクセス権限を一貫して管理するために IAM が使用されるようになりました。既に多数のサービスがリソース・グループおよび IAM の使用を採用しており、時間がたつにつれて、さらに多くのサービスが、リソースの編成とアクセス権限の管理のためにこの新しい方法に移行するでしょう。

リソース・グループと Cloud Foundry の組織およびスペースとの主な違いは、アクセス制御およびアカウント・リソース編成です。リソース・グループは、IAM ポリシーを使用してアクセス制御される IAM 対応サービスをアカウント内で編成します。組織およびスペースは、アクセス制御のために Cloud Foundry 役割を使用して管理され、Cloud Foundry リソースはスペースに割り当てられます。組織およびスペースは、Cloud Foundry レルム内のリソースに対してのみ、編成およびアクセス管理のために使用できます。一方、リソース・グループおよび IAM は、{{site.data.keyword.Bluemix_notm}} 全体で複数のタイプのリソースに対して使用できます。
