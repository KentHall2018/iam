# 使用者角色及許可權
{: #userroles}

您可以從帳戶的**使用者**頁面，跨 {{site.data.keyword.Bluemix_notm}} 平台及「基礎架構」服務來管理使用者。若要存取「使用者」頁面，請從 {{site.data.keyword.Bluemix_notm}} 功能表中按一下**管理** &gt; **帳戶** &gt; **使用者**。帳戶擁有者可以執行用來管理使用者、許可權、組織及空間的所有作業。Cloud Foundry 組織管理員及空間管理員也具有管理使用者許可權的存取權，而使用者已新增至其每一個所管理的組織及空間。
{:shortdesc}

如果您是新增至另一人之帳戶的使用者，而且要檢視獲指派的角色及許可權，請移至**管理** &gt; **安全** &gt; **身分及存取** &gt; **使用者**，然後按一下您的名稱。

## 帳戶角色
{: #userrolesinfo}

在帳戶層次，有兩種角色可以存取不同的帳戶管理特性：

| 帳戶角色| 許可權|
|----------------|---------|
|擁有者| 帳戶擁有者可以存取其設定檔、使用者、計費資訊、消費通知及用量儀表板。從「使用者」頁面中，擁有者可以邀請新的使用者，以及調整角色。擁有者也可以新增促銷額度、設定或變更計費限制、設定服務存取，以及管理組織和空間。|
|成員| 成員可以在 {{site.data.keyword.Bluemix_notm}} 標頭中存取其設定檔、顯示帳戶裡作用中使用者的「使用者」頁面，以及帳戶額度和計費限制。|
{:caption="表 1. 帳戶角色及許可權" caption-side="top"}

所有新的使用者都會新增為帳戶的成員。您可以指派受邀者的組織和空間角色，以啟用 {{site.data.keyword.Bluemix_notm}} 中的特定視圖和許可權。您的受邀者接受邀請並加入 {{site.data.keyword.Bluemix_notm}} 之後，您就可以在「使用者」頁面上編輯其角色。

## Cloud Foundry 角色
{: #cfroles}

Cloud Foundry 角色包括帳戶內所定義之組織及空間的存取許可權。Cloud Foundry 角色不會啟用使用者許可權，以在服務的環境定義內完成動作。您可以在組織層次指派下列角色：

| 組織角色| 許可權|
|-------------------|-------------|
|管理員| 組織管理員可以建立、檢視、編輯或刪除組織內的空間、檢視組織的用量和配額、邀請使用者加入組織、管理誰可以存取組織及其在組織中的角色，以及管理組織的自訂網域。|
|帳單管理員| 帳單管理員可以在「用量儀表板」頁面上檢視組織的運行環境及服務用量資訊。|
|審核員| 組織審核員可以檢視組織中的應用程式及服務內容。審核員也可以檢視組織中的使用者及其獲指派的角色，以及組織的配額。|
{:caption="表 2. 組織角色及許可權" caption-side="top"}

您可以在空間層次指派下列角色：

| 空間角色| 許可權|
|------------|-------------|
|管理員| 空間管理員可以在空間內新增現有使用者以及管理角色。空間管理員也可以檢視空間中每一個應用程式的實例數、服務連結及資源用量。|
|開發人員| 空間開發人員可以建立、刪除及管理空間內的應用程式和服務。部分管理作業包括部署應用程式、啟動或停止應用程式、重新命名應用程式、刪除應用程式、重新命名空間、將服務連結至應用程式或取消服務與應用程式的連結，以及檢視空間中每一個應用程式的實例數、服務連結及資源用量。此外，空間開發人員可以建立內部或外部 URL 與空間中應用程式的關聯。|
|審核員| 空間審核員具有所有空間資訊（例如空間中每一個應用程式的實例數、服務連結及資源用量的相關資訊）的唯讀存取權。|
{:caption="表 3. 空間角色及許可權" caption-side="top"}

**附註**：獲指派管理員或開發人員空間角色的使用者可以存取 VCAP_SERVICES 環境變數。不過，獲指派審核員角色的使用者無法存取 VCAP_SERVICES。

## 身分及存取管理原則和角色
{: #iamusermanpol}

帳戶擁有者會自動獲指派帳戶管理者角色來進行「身分及存取管理 (IAM)」，這可讓您指派及管理服務原則。服務原則可以指派給個別使用者或服務 ID，而且所指派的原則可以定義整個實例或個別實例層次的存取權。

### 服務原則

原則會使用屬性組合來定義適用的資源集，將對資源集的一或多個角色指派給使用者或服務 ID。當您指派原則時，請先指定服務。然後，您可以選取所要指派的一或數個角色。視您選取的服務而定，可能還有其他配置選項可供使用。

如果您有適當的角色，可以指派及管理原則。下表顯示原則管理作業，以及各項作業所需的角色。

| 動作| 所需的角色|
|----------|---------|
| 建立帳戶對所有服務及實例的原則| 帳戶管理者|
| 建立帳戶中之服務的原則| 帳戶管理者，或者帳戶中服務的管理者|
| 建立服務實例| 帳戶管理者，或者帳戶中服務的管理者或編輯者|
| 建立服務實例的原則| 帳戶管理者，或帳戶中服務的管理者，或服務實例的管理者|
{: caption="表 4. 用來管理「已啟用 IAM 的服務」原則的管理作業" caption-side="top"}

### 服務原則角色
{: #iamusermanrol}

IAM 容許您管理及定義帳戶中之使用者與資源的存取權。如果您使用的服務可以使用 IAM 進行使用者存取控制的方式來管理，則您的帳戶內會有允許不同動作的兩種角色：平台管理和服務存取角色。

<dl>
<dt>平台管理角色</dt>
<dd>這些角色適用於可以使用 IAM 進行使用者存取控制來管理的所有服務，包括管理者、編輯者、操作員、檢視者及帳單管理者。這些角色對映到可以在平台層次對 {{site.data.keyword.Bluemix_notm}} 資源執行的動作。例如，這些動作中有部分是能夠建立實例、將實例連接至應用程式、管理服務 ID、管理使用者和許可權，以及管理帳戶的計費與配額等能力。</dd>
<dt>服務存取角色</dt>
<dd>這些角色是服務所特有。管理員、撰寫者和讀者角色會定義使用者使用服務及執行服務 API 呼叫的能力。因為每個服務會定義具有給定角色的使用者能執行的動作，所以服務可能不會使用本文件中所述的所有可用角色。</dd>
</dl>

**附註**：在使用者介面中指派原則時，您可能不會看到所有角色都列為選項，因為只有適用於您在原則裡選取之服務的角色會顯示。如需哪些角色已啟用以及每個存取角色針對每個服務允許哪些動作的特定資訊，請參閱該服務的文件。


#### 平台管理角色

平台管理角色讓使用者能被指派不同的許可權層次，以便執行平台動作。除了主控台中提供的角色說明之外，下表還提供獲指派各角色之使用者可以執行的部分平台管理動作範例。

| 平台管理角色| 使用者可以對帳戶中的服務執行的動作| 服務 ID 的動作|
|:-----------------|:--------------|:---------------|
| 檢視者| 檢視實例| 檢視 ID 及 API 金鑰|
| 操作員|  檢視及連結服務實例| 不適用|
| 編輯者|  建立、刪除、編輯、暫停、繼續、檢視、連結服務實例| 刪除 ID 及建立與刪除 API 金鑰|
| 管理者|  服務的所有管理動作| 建立及刪除 ID 與 API 金鑰、指派原則給 ID|
{: caption="表 5. 平台管理角色及動作範例" caption-side="top"}

部分服務可能將特定的動作對映至與服務管理相關的平台管理角色，而不是與服務存取相關的平台管理角色。例如，請參閱下表，下表詳述對映至這些角色的 {{site.data.keyword.containershort_notm}} 服務動作。


| 平台管理角色| 動作的說明| {{site.data.keyword.containershort_notm}} 服務的動作範例|
|:-----------------|:-----------------|:-----------------|
| 檢視者| 可以檢視服務實例，但不能修改| <ul><li>列出叢集</li><li>檢視叢集的詳細資料</li></ul>|
| 編輯者| 執行所有平台動作，但管理帳戶和指派存取原則除外|<ul><li>將服務連結至叢集</li><li>建立 Webhook</li></ul> |
| 操作員| 執行配置及操作服務實例所需的平台動作，例如檢視服務的儀表板。| <ul><li>新增或移除工作者節點</li><li>重新開機或重新載入工作者節點</li><li>將服務連結至叢集</li></ul> |
| 管理者| 根據指派給這個角色的資源執行所有平台動作，包括指派存取原則給其他使用者。|<ul><li>移除叢集</li><li>建立叢集</li><li>更新使用者存取原則</li><li>檢視者、編輯者及操作員可以執行的所有動作</li></ul>|
{: caption="表 6. 平台管理角色及動作範例" caption-side="top"}


#### 服務存取角色

服務存取角色讓使用者能被指派不同的許可權層次，以便呼叫服務的 API。下表提供可以根據指派的服務存取角色且根據使用 {{site.data.keyword.objectstorageshort}} 服務採取的範例動作。

**附註**：可以針對每個已指派角色採取的動作，會視您為原則選取的服務而不同。

| 服務存取角色| 動作的說明| {{site.data.keyword.objectstorageshort}} 服務的動作範例|
|:-----------------|:-----------------|:-----------------|
|  讀者| 在服務內執行唯讀動作，例如檢視服務特有的資源| 列出及下載物件|
| 撰寫者| 撰寫者具有超越讀者角色的許可權，包括建立及編輯服務特有的資源。| 建立及破壞儲存區與物件|
| 管理員| 管理員具有超越撰寫者角色的許可權，以完成服務所定義的特許動作。此外，您也可以建立及編輯服務特有的資源。| 管理儲存空間的所有層面、建立及破壞儲存區與物件|
{: caption="表 7. 服務存取使用者角色及動作範例" caption-side="top"}


## 基礎架構許可權
{: #infrapermissions}

您可以在邀請使用者時設定下列起始許可權：

| 許可權集 | 動作的說明|
|---------------------------|------------------------|
|僅限檢視| 具有此許可權的使用者只能檢視系統內的項目。|
|基本使用者| 具有此許可權的使用者可以在系統內執行基本動作，例如新增問題單及管理裝置。|
|超級使用者| 具有此許可權的使用者可以在系統中執行所有可用動作。|
{:caption="表 8. 基礎架構許可權" caption-side="top"}

使用者接受邀請之後，可以設定其他許可權。邀請時所設定的起始許可權未授與裝置的存取權，因此，在使用者接受邀請之後，您必須在「控制入口網站」中授與裝置存取權。
