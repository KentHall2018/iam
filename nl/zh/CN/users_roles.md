# IAM 访问权
{: #userroles}

在帐户中组织成资源组的所有服务均使用 {{site.data.keyword.Bluemix_notm}} Identity and Access Management (IAM) 进行管理。将自动为帐户所有者分配 Cloud IAM 的帐户管理员角色。作为帐户管理员，您可以为用户分配和管理访问权，创建资源组，供应服务实例，以及执行可以通过分配 Cloud IAM 角色来委派的其他所有任务。可以通过创建策略来为用户或服务标识设置要访问的目标以及设置用于定义允许的访问权类型的角色，从而为用户和服务标识提供访问权。


## 什么是 Cloud IAM 策略？谁可以分配这些策略？
{: #iamusermanpol}

可以通过创建策略来为用户或服务标识设置要访问的目标以及设置用于定义允许的访问权类型的角色，从而为用户和服务标识提供访问权。策略使用属性组合来定义适用的一组资源，从而为用户或服务标识授予对该组资源的一个或多个角色。分配策略时，首先选择是为资源组还是单个资源设置策略。然后，根据初始选择，可以选择资源组中的服务或所选资源的单个实例。根据选择的服务，可能会提供更多配置选项。最后，可以选择要分配的一个或多个角色。 

如果您具有合适的角色，那么可以分配和管理策略。下表显示了策略管理任务以及每个任务所需的角色。

| 操作| 所需角色 |
|----------|---------|
| 在帐户中为所有服务和实例创建策略| 帐户所有者或帐户中所有服务的管理员| 
| 创建关于帐户中某个服务的策略| 帐户所有者或者帐户中该服务的管理员|
| 创建关于某个服务实例的策略| 帐户所有者、帐户中该服务的管理员、相关资源组中所有服务的管理员或该服务实例的管理员|
| 创建用于管理资源组的策略| 帐户所有者或资源组的管理员|
{: caption="表 1. 允许创建访问策略的用户" caption-side="top"} 


## Cloud IAM 角色
{: #iamusermanrol}

通过 Cloud IAM，您可以为帐户中的用户和资源管理和定义访问权。如果可以使用 IAM 来管理您所使用的服务，那么可以分配两种类型的角色：平台管理角色和服务访问角色。

<dl>
<dt>平台管理角色</dt> 
<dd>平台管理角色涵盖一系列操作，包括创建实例、管理服务标识、管理用户和许可权以及创建资源组的能力。最常见的平台角色是管理员、编辑者、操作员和查看者。</dd>
<dt>服务访问角色</dt>
<dd>服务访问角色定义用户或服务对服务实例执行操作的能力，例如访问 UI 或执行 API 调用。有三种可能的角色：管理员、作者和读者。</dd>
</dl> 

在 UI 中分配策略时，您可能看不到所有的角色都作为选项列出，这是因为仅会显示可用于您在策略中所选服务的角色。有关启用哪些角色以及每个访问角色允许对每个服务执行的操作的特定信息，请参阅该服务的相关文档。
{: tip}

在单个访问策略中使用这些角色的组合，可以通过分配对以下任一项的访问权来为用户和服务标识提供细颗粒度访问权：

* 帐户内的所有资源
* 属于单个资源组的所有服务内的所有资源以及管理资源组的能力
* 资源组中单个服务内的所有资源，以及管理资源组的能力
* 整个帐户中单个服务内的所有资源，而不考虑将其分配到的资源组
* 单个实例中的资源
* 实例内的单个资源类型，例如 {{site.data.keyword.objectstorageshort}} 实例中的存储区

要使另一个用户具有对帐户的完全访问权，以便管理用户访问权和管理所有帐户资源（包括创建资源组的能力），请通过选择**所有启用“身份和访问权”的服务**并分配**管理员**角色来设置策略，以授予用户对帐户中所有资源的访问权。
{: tip}

### 平台管理角色

通过平台管理角色，用户可以分配有各种不同级别的许可权，以在帐户内执行平台操作。除了控制台中提供的角色的描述外，下表还提供了分配有每个角色的用户可以执行的一些平台管理操作示例。您可以参阅每个服务的文档，以了解这些角色如何在所使用的服务上下文中应用于用户。

| 平台管理角色 | 用户在帐户中可以对服务执行的操作 | 服务标识的操作 | 用于访问资源组的操作| 对资源组中资源的操作|
|:-----------------|:--------------|:---------------|:----------------|:-----------------|
| 查看者| 查看实例、别名、绑定和凭证| 查看标识和 API 密钥 | 查看资源组| 仅查看资源组中指定的实例|
| 操作员|  查看实例并管理别名、绑定和凭证| 不适用 | 不适用 | 不适用 |
| 编辑者|  创建、删除、编辑和查看实例。管理别名、绑定和凭证| 创建和删除标识及 API 密钥| 查看和编辑资源组的名称| 仅对资源组中指定的实例执行创建、删除、编辑、暂挂、恢复、查看和绑定操作|
| 管理员|  服务的所有管理操作 | 创建和删除标识和 API 密钥、向标识分配策略 | 查看、编辑和管理对资源组的访问权| 对资源组中指定实例的所有管理操作|
{: caption="表 2. 平台管理角色和操作的示例" caption-side="top"}

一些服务可能将特定操作映射到与服务管理而非与服务访问相关的平台管理角色。作为示例，请查看下表，其详细说明了映射到这些角色的 {{site.data.keyword.containershort_notm}} 服务操作。


| 平台管理角色 | 操作描述| {{site.data.keyword.containershort_notm}} 的操作示例 |
|:-----------------|:-----------------|:-----------------|
| 查看者| 可以查看服务实例，但是不能修改它们 | <ul><li>列出集群</li><li>查看集群的详细信息</li></ul>|
| 编辑者| 执行除管理帐户和分配访问策略之外的所有其他平台操作 |<ul><li>将服务绑定到集群</li><li>创建 Webhook</li></ul> |
| 操作员| 执行配置和操作服务实例所需的平台操作，如查看服务的仪表板。| <ul><li>添加或除去工作程序节点</li><li>重新引导或重新装入工作程序节点</li><li>将服务绑定到集群</li></ul> |
| 管理员| 基于分配给此角色的资源执行所有平台操作，包括向其他用户分配访问策略。|<ul><li>除去集群</li><li>创建集群</li><li>更新用户访问策略</li><li>查看者、编辑者和操作员可以执行的所有操作</li></ul>|
{: caption="表 3. {{site.data.keyword.containershort_notm}} 服务的平台管理角色和操作的示例" caption-side="top"}


### 服务访问角色

服务访问角色可使用户分配有不同级别的许可权，以调用服务 API 和访问服务的 UI。下表提供根据所分配的角色，可使用 {{site.data.keyword.objectstorageshort}} 服务执行的操作示例。

**注**：每个所分配角色可采取的操作根据您为策略所选择的服务而有所不同。

| 服务访问角色 | 操作描述| {{site.data.keyword.objectstorageshort}} 服务的操作示例 |
|:-----------------|:-----------------|:-----------------|
|  读者 | 在服务内执行只读操作，如查看服务特定资源 | 列出和下载对象 |
| 作者 | 作者的许可权高于读者角色，包括创建和编辑服务特定资源。| 创建和销毁存储区和对象 |
| 管理员| 管理者的许可权高于作者角色，可完成服务所定义的特权操作。此外，还可以创建和编辑服务特定资源。| 管理数据存储的所有方面、创建和销毁存储区和对象 |
{: caption="表 4. 服务访问用户角色和操作的示例" caption-side="top"}
